# ResourcePackManager — پراکسی پلگ ان

> **⚠ یہ پلگ ان صرف Geyser + Floodgate چلانے والے سرورز کے لیے مفید ہے۔**
> اس کا واحد مقصد ہر بیک اینڈ کے Bedrock میں تبدیل کیے گئے resource pack کو
> ضم کرنا اور Geyser کے ذریعے Bedrock کلائنٹس کو فراہم کرنا ہے۔ **Java Edition
> کھلاڑیوں کو اس پلگ ان کی ضرورت نہیں** — نیٹ ورک سیٹ اپ میں Java pack کی
> فراہمی براہ راست انفرادی بیک اینڈز کے ذریعے ہوتی ہے۔ اگر آپ کا نیٹ ورک صرف
> Java ہے (Geyser کے ذریعے آنے والا کوئی Bedrock کھلاڑی نہیں)، تو آپ اس فولڈر
> کو مکمل طور پر نظر انداز کر سکتے ہیں۔

اس فولڈر میں **RSPM پراکسی پلگ ان** کی JAR فائلیں موجود ہیں۔ یہ آپ کے **پراکسی**
(Velocity / BungeeCord / Waterfall) پر جاتی ہیں، Minecraft بیک اینڈ سرور پر **نہیں**۔
پراکسی پلگ ان ہر بیک اینڈ کے Bedrock میں تبدیل کیے گئے resource pack کو ایک ہی pack
میں ضم کرتا ہے اور Geyser کے ذریعے Bedrock کلائنٹس کو فراہم کرتا ہے۔

اگر آپ ایک اسٹینڈ ایلون سرور چلا رہے ہیں (سامنے کوئی پراکسی نہیں)، تو آپ اس فولڈر
کو مکمل طور پر نظر انداز کر سکتے ہیں۔ JAR ہر بوٹ پر بہرحال نکالی جاتی ہیں، لہٰذا
اگر آپ بعد میں کبھی پراکسی شامل کریں تو وہ تیار رہتی ہیں۔

---

## مجھے کون سی JAR استعمال کرنی چاہیے؟

اپنے پراکسی سافٹ ویئر کے مطابق بالکل **ایک** کا انتخاب کریں:

| پراکسی سافٹ ویئر          | یہ JAR استعمال کریں                       |
|--------------------------|------------------------------------------|
| Velocity                 | `ResourcePackManager-Velocity.jar`       |
| BungeeCord               | `ResourcePackManager-BungeeCord.jar`     |
| Waterfall (Bungee فورک)  | `ResourcePackManager-BungeeCord.jar`     |

دونوں انسٹال نہ کریں — صرف وہی جو میل کھاتا ہے۔ دونوں انسٹال کرنے سے غلط JAR لوڈ
کرنے کی کوشش کرنے والے پلیٹ فارم پر بوٹ کی غلطیاں ہوں گی۔

---

## انسٹالیشن — 2 مراحل

### 1. JAR کو پراکسی ہوسٹ پر کاپی کریں

متعلقہ JAR کو **اس** فولڈر سے اپنے پراکسی کی `plugins/` ڈائریکٹری میں کاپی کریں۔
عام طریقے:

- **ایک ہی مشین**: drag-and-drop، یا `cp` / `copy`۔
- **ریموٹ پراکسی**: `scp ResourcePackManager-Velocity.jar user@proxy-host:/path/to/proxy/plugins/`
- **ہوسٹڈ پراکسی پینل** (Pterodactyl وغیرہ): پینل کے فائل مینیجر کے ذریعے اپ لوڈ کریں۔

### 2. پراکسی کو دوبارہ شروع کریں

بس اتنا ہی۔ کوئی کنفگ ترمیم کرنے یا کوئی key پیسٹ کرنے کی ضرورت نہیں۔

پلگ ان بوٹ پر پراکسی کی `plugins/floodgate/key.pem` فائل پڑھتا ہے اور اس فائل
سے خود بخود نیٹ ورک شناخت اخذ کرتا ہے۔ چونکہ Floodgate کو ویسے بھی درکار ہے کہ
یہ فائل ہر بیک اینڈ **اور** پراکسی پر یکساں ہو (Bedrock تصدیق کے لیے ضروری
ہے)، اخذ کردہ key خود بخود ہر بیک اینڈ کے ساتھ مماثلت رکھتا ہے۔

پہلا ضم عام طور پر تمام بیک اینڈز کے آن لائن ہونے کے ~10 سیکنڈ کے اندر مکمل
ہو جاتا ہے۔

## تصدیق کریں کہ یہ کام کر رہا ہے

**پراکسی کنسول** (دوبارہ شروع کرنے کے ~10s کے اندر، یہ فرض کرتے ہوئے کہ
بیک اینڈز چل رہے ہیں):

```
[ResourcePackManager] Merged pack ready at .../merged/Bedrock.zip (sha1 ...)
[ResourcePackManager] Geyser mappings deployed to .../custom_mappings
[ResourcePackManager] ✔ Network resource pack is now ready (... KB, sha1ABCD1234)
```

**Bedrock کلائنٹ**: Bedrock کے ذریعے کنیکٹ کریں۔ دنیا میں پہنچنے سے پہلے آپ کو
resource pack ڈاؤن لوڈ کا اشارہ نظر آنا چاہیے۔ کسٹم آئٹمز سادے armor stands کی بجائے
اپنے مطلوبہ ماڈلز کے ساتھ رینڈر ہوتے ہیں۔

**بیک اینڈ پر `/rspm status`**: pack کی حالت، ہوسٹنگ موڈ، اور network-key کا
فنگرپرنٹ دکھاتا ہے۔ دونوں اطراف کے منسلک ہونے کی تصدیق کے لیے آخری 4 حروف کو
پراکسی کنفگ سے ملائیں۔

## عام مسائل

### پراکسی بوٹ پر "Floodgate key.pem missing"

پراکسی پلگ ان کو `plugins/floodgate/key.pem` نہیں ملی اور یہ غیر فعال رہا۔ حل:

1. پراکسی پر **Floodgate انسٹال کریں**۔ Bedrock کھلاڑیوں کو پراکسی تک پہنچنے
   کے لیے یہ ویسے بھی درکار ہے، لہٰذا RSPM کے بغیر بھی اس کی ضرورت ہے۔
2. یقینی بنائیں کہ پراکسی پر `plugins/floodgate/key.pem` ہر بیک اینڈ پر موجود
   اسی فائل کے ساتھ **بائٹ بہ بائٹ یکساں** ہے۔ Floodgate ڈیفالٹ کے طور پر ہر
   انسٹال کے لیے مختلف key بناتا ہے — کسی ایک بیک اینڈ سے ایک معیاری
   `key.pem` تمام دوسرے components (دیگر بیک اینڈز + پراکسی) پر کاپی کریں،
   پھر سب کچھ دوبارہ شروع کریں۔ Floodgate کو Bedrock تصدیق کے لیے ویسے بھی یہ
   درکار ہے، تو اگر آپ کے نیٹ ورک پر Bedrock کھلاڑی فی الحال کام کر رہے ہیں،
   تو یہ پہلے سے ہو چکا ہے۔

### Bedrock کنیکٹ ہوتا ہے لیکن کسٹم ماڈلز نظر نہیں آتے

سب سے عام وجہ: پراکسی بیک اینڈ کے پہلا Bedrock pack بنانے سے **پہلے** شروع ہو
گیا۔ Geyser صرف بوٹ پر کسٹم آئٹمز کو رجسٹر کرتا ہے — ایک بار خالی mappings فائل
کے ساتھ چلنے کے بعد، یہ ویسا ہی رہتا ہے۔ بیک اینڈ کے
`Wrote merged Geyser mappings: N entries` لاگ کرنے کے بعد **پراکسی کو دوبارہ
شروع کریں**۔ بعد کے ضم Geyser کے pack سرونگ راستے کے ذریعے خود بخود اٹھا
لیے جاتے ہیں، لیکن کسٹم آئٹم ٹیبل بوٹ پر طے ہو جاتی ہے۔

### پراکسی بوٹ پر "Duplicate bedrock_identifier" انتباہات

دو بیک اینڈز نے ایک ہی بنیادی آئٹم کے لیے ایک ہی Bedrock identifier جاری کیا۔
آخری لکھنے والا جیتتا ہے؛ اگر آپ کو وہ آئٹم فراہم کرنے کے لیے صرف ایک بیک اینڈ
درکار ہے تو یہ بے ضرر ہے۔ اگر دونوں بیک اینڈز کو ایک ہی بنیادی آئٹم کے تحت
**مختلف** کسٹم آئٹمز ہوسٹ کرنے چاہئیں، تو آپ کے پاس ایک حقیقی تصادم ہے —
سورس Java ماڈلز میں سے ایک کا نام تبدیل کریں تاکہ خود بخود تیار کردہ hashes
مختلف ہوں۔

### RSPM اپ ڈیٹ کرنا

بیک اینڈ RSPM JAR کو اپ ڈیٹ کرنے کے بعد، اس فولڈر سے متعلقہ
`ResourcePackManager-Velocity.jar` / `-BungeeCord.jar` کو بھی پراکسی پر
**دوبارہ کاپی** کریں اور پراکسی کو دوبارہ شروع کریں۔ بیک اینڈ ہر بوٹ پر ان
کو دوبارہ تیار کرتا ہے، اس لیے وہ ہمیشہ بیک اینڈ ورژن کے ساتھ سنک میں رہتی ہیں۔
