# ResourcePackManager — প্রক্সি প্লাগইন

> **⚠ এই প্লাগইন শুধুমাত্র Geyser + Floodgate চালানো সার্ভারগুলির জন্য উপযোগী।**
> এর একমাত্র উদ্দেশ্য হল প্রতিটি ব্যাকএন্ডের Bedrock-এ রূপান্তরিত resource pack
> একত্রিত করা এবং Geyser-এর মাধ্যমে Bedrock ক্লায়েন্টদের কাছে পৌঁছে দেওয়া।
> **Java Edition খেলোয়াড়দের এই প্লাগইন প্রয়োজন নেই** — নেটওয়ার্ক সেটআপে
> Java pack বিতরণ সরাসরি স্বতন্ত্র ব্যাকএন্ডগুলো দ্বারা পরিচালিত হয়। যদি আপনার
> নেটওয়ার্ক শুধুমাত্র Java হয় (Geyser-এর মাধ্যমে আসা কোনো Bedrock খেলোয়াড়
> না থাকে), আপনি এই ফোল্ডারটি সম্পূর্ণরূপে উপেক্ষা করতে পারেন।

এই ফোল্ডারে **RSPM প্রক্সি প্লাগইন**-এর JAR ফাইলগুলো রয়েছে। এগুলো আপনার
**প্রক্সি** (Velocity / BungeeCord / Waterfall)-এ যাবে, ব্যাকএন্ড Minecraft সার্ভারে
**নয়**। প্রক্সি প্লাগইন প্রতিটি ব্যাকএন্ডের রূপান্তরিত Bedrock resource pack-কে
একটি একক pack-এ মিলিয়ে নেয় এবং Geyser-এর মাধ্যমে Bedrock ক্লায়েন্টদের কাছে
সরবরাহ করে।

আপনি যদি একটি স্বতন্ত্র সার্ভার চালান (সামনে কোনো প্রক্সি নেই), তাহলে এই ফোল্ডারটি
সম্পূর্ণরূপে উপেক্ষা করতে পারেন। JAR-গুলো প্রতিটি বুটে যাই হোক না কেন এক্সট্র্যাক্ট
করা হয়, তাই পরে কখনো প্রক্সি যোগ করলে সেগুলো প্রস্তুত থাকে।

---

## আমি কোন JAR ব্যবহার করব?

আপনার প্রক্সি সফটওয়্যার অনুযায়ী ঠিক একটি বাছাই করুন:

| প্রক্সি সফটওয়্যার          | এই JAR ব্যবহার করুন                        |
|--------------------------|------------------------------------------|
| Velocity                 | `ResourcePackManager-Velocity.jar`       |
| BungeeCord               | `ResourcePackManager-BungeeCord.jar`     |
| Waterfall (Bungee ফর্ক)  | `ResourcePackManager-BungeeCord.jar`     |

দুটোই ইনস্টল করবেন না — শুধু মেলানোটাই। দুটোই ইনস্টল করলে যে প্ল্যাটফর্ম ভুল JAR
লোড করার চেষ্টা করবে সেখানে বুট ত্রুটি ঘটবে।

## ইনস্টলেশন — ২ ধাপ

### ১. JAR প্রক্সি হোস্টে কপি করুন

প্রাসঙ্গিক JAR-টি **এই** ফোল্ডার থেকে আপনার প্রক্সির `plugins/` ডিরেক্টরিতে কপি
করুন। সাধারণ পদ্ধতি:

- **একই মেশিন**: drag-and-drop, অথবা `cp` / `copy`।
- **দূরবর্তী প্রক্সি**: `scp ResourcePackManager-Velocity.jar user@proxy-host:/path/to/proxy/plugins/`
- **হোস্টেড প্রক্সি প্যানেল** (Pterodactyl ইত্যাদি): প্যানেলের ফাইল ম্যানেজার
  দিয়ে আপলোড করুন।

### ২. প্রক্সি পুনরায় চালু করুন

ব্যস। কোনো কনফিগ সম্পাদনা করার বা কোনো key পেস্ট করার দরকার নেই।

প্লাগইন বুটে প্রক্সির `plugins/floodgate/key.pem` পড়ে এবং সেই ফাইল থেকে
স্বয়ংক্রিয়ভাবে নেটওয়ার্ক পরিচয় বের করে। যেহেতু Floodgate-এর এমনিতেই প্রয়োজন
যে এই ফাইল প্রতিটি ব্যাকএন্ডে **এবং** প্রক্সিতে এক হয় (Bedrock প্রমাণীকরণ
কাজ করার জন্য), নির্ণীত key স্বয়ংক্রিয়ভাবে প্রতিটি ব্যাকএন্ডের সাথে মেলে।

প্রথম মার্জ সাধারণত সব ব্যাকএন্ড চালু হওয়ার ~১০ সেকেন্ডের মধ্যে সম্পন্ন হয়।

## কাজ করছে কিনা যাচাই করা

**প্রক্সি কনসোল** (পুনরায় চালুর ~১০s-এর মধ্যে, ধরে নিচ্ছি ব্যাকএন্ডগুলো চলছে):

```
[ResourcePackManager] Merged pack ready at .../merged/Bedrock.zip (sha1 ...)
[ResourcePackManager] Geyser mappings deployed to .../custom_mappings
[ResourcePackManager] ✔ Network resource pack is now ready (... KB, sha1ABCD1234)
```

**Bedrock ক্লায়েন্ট**: Bedrock দিয়ে সংযোগ করুন। বিশ্বে পৌঁছানোর আগে আপনার
resource pack ডাউনলোডের প্রম্পট দেখা উচিত। কাস্টম আইটেমগুলো সাধারণ armor
stand-এর বদলে তাদের নির্ধারিত মডেল সহ রেন্ডার হয়।

**ব্যাকএন্ডে `/rspm status`**: pack-এর অবস্থা, হোস্টিং মোড এবং network-key-এর
ফিঙ্গারপ্রিন্ট দেখায়। দুই দিকই সংযুক্ত আছে নিশ্চিত করতে শেষ ৪টি অক্ষরের সাথে
প্রক্সির কনফিগ মেলান।

## সাধারণ সমস্যা

### প্রক্সি বুটে "Floodgate key.pem missing"

প্রক্সি প্লাগইন `plugins/floodgate/key.pem` খুঁজে পায়নি এবং নিষ্ক্রিয় রয়েছে। সমাধান:

1. প্রক্সিতে **Floodgate ইনস্টল করুন**। Bedrock খেলোয়াড়দের প্রক্সিতে পৌঁছাতে
   এটি যেকোনো ভাবে দরকার, তাই RSPM ছাড়াও এটি প্রয়োজন।
2. নিশ্চিত করুন যে প্রক্সিতে `plugins/floodgate/key.pem` প্রতিটি ব্যাকএন্ডের
   একই ফাইলের সাথে **বাইট-বাই-বাইট অভিন্ন**। Floodgate ডিফল্টভাবে প্রতিটি
   ইনস্টলের জন্য আলাদা key তৈরি করে — যেকোনো ব্যাকএন্ড থেকে একটি প্রামাণিক
   `key.pem` অন্য সব components (অন্য ব্যাকএন্ড + প্রক্সি)-এ কপি করুন, তারপর
   সব কিছু পুনরায় চালু করুন। Floodgate-এর Bedrock প্রমাণীকরণের জন্য এটি
   ইতিমধ্যেই প্রয়োজন, তাই যদি আপনার নেটওয়ার্কে Bedrock খেলোয়াড়রা বর্তমানে
   কাজ করে, তাহলে এটি ইতিমধ্যেই করা হয়েছে।

### Bedrock সংযোগ হয় কিন্তু কাস্টম মডেল দেখায় না

সবচেয়ে সাধারণ কারণ: ব্যাকএন্ড তার প্রথম Bedrock pack তৈরি করার **আগেই**
প্রক্সি চালু হয়েছে। Geyser কাস্টম আইটেম শুধু বুটে রেজিস্টার করে — একবার
এটি খালি mappings ফাইল নিয়ে চলতে শুরু করলে, ওভাবেই থাকে। ব্যাকএন্ড
`Wrote merged Geyser mappings: N entries` লগ করার পরে **প্রক্সি পুনরায়
চালু করুন**। পরবর্তী মার্জগুলো Geyser-এর pack-সার্ভিং রুট দ্বারা
স্বয়ংক্রিয়ভাবে গৃহীত হয়, কিন্তু কাস্টম-আইটেম টেবিল বুটেই স্থির হয়ে যায়।

### প্রক্সি বুটে "Duplicate bedrock_identifier" সতর্কতা

দুটো ব্যাকএন্ড একই বেস আইটেমের জন্য একই Bedrock identifier তৈরি করেছে।
শেষ লেখক জেতে; যদি ওই আইটেম সরবরাহের জন্য আপনার শুধু একটি ব্যাকএন্ড
দরকার হয় তবে এটা ক্ষতিকর নয়। যদি দুটি ব্যাকএন্ডের একই বেস আইটেমের অধীনে
**ভিন্ন** কাস্টম আইটেম হোস্ট করা উচিত, তাহলে আপনার একটি প্রকৃত সংঘর্ষ আছে —
উৎস Java মডেলগুলোর একটি নাম পরিবর্তন করুন যাতে স্বয়ংক্রিয়ভাবে তৈরি হ্যাশ
ভিন্ন হয়।

### RSPM আপডেট করা

ব্যাকএন্ড RSPM JAR আপডেট করার পরে, এই ফোল্ডার থেকে সংশ্লিষ্ট
`ResourcePackManager-Velocity.jar` / `-BungeeCord.jar` প্রক্সিতে **পুনরায়
কপি** করুন এবং প্রক্সি পুনরায় চালু করুন। ব্যাকএন্ড প্রতিটি বুটে এগুলো পুনঃ
উৎপন্ন করে, তাই এগুলো সর্বদা ব্যাকএন্ড সংস্করণের সাথে সিঙ্কে থাকে।
