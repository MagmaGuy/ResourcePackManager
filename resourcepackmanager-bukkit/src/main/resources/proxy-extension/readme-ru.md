# ResourcePackManager — Плагин прокси

> **⚠ Этот плагин полезен только для серверов, использующих Geyser + Floodgate.**
> Его единственная задача — собрать преобразованные Bedrock-паки от каждого
> бэкенда и раздать их Bedrock-клиентам через Geyser. **Игрокам Java Edition
> этот плагин не нужен** — раздачей паков для Java в сетевых конфигурациях
> занимаются непосредственно отдельные бэкенды. Если ваша сеть только Java
> (без Bedrock-игроков, идущих через Geyser), эту папку можно полностью
> игнорировать.

В этой папке находятся JAR-файлы **плагина прокси RSPM**. Они устанавливаются
на ваш **прокси** (Velocity / BungeeCord / Waterfall), а **НЕ** на бэкенд-сервер
Minecraft. Плагин прокси сливает преобразованные Bedrock-паки от всех бэкендов в
один общий пак и раздаёт его Bedrock-клиентам через Geyser.

Если вы держите автономный сервер (без прокси перед ним), эту папку можно
полностью игнорировать. JAR'ы всё равно распаковываются при каждом старте, так
что они будут готовы, если вы решите добавить прокси позже.

---

## Какой JAR использовать?

Выберите ровно **один** в зависимости от вашего ПО прокси:

| ПО прокси                | Используйте этот JAR                     |
|--------------------------|------------------------------------------|
| Velocity                 | `ResourcePackManager-Velocity.jar`       |
| BungeeCord               | `ResourcePackManager-BungeeCord.jar`     |
| Waterfall (форк Bungee)  | `ResourcePackManager-BungeeCord.jar`     |

Не устанавливайте оба — только подходящий. Установка обоих вызовет ошибки старта
на платформе, которая попытается загрузить неподходящий JAR.

## Установка — 2 шага

### 1. Скопируйте JAR на хост прокси

Скопируйте нужный JAR из **этой** папки в каталог `plugins/` вашего прокси.
Типичные способы:

- **Та же машина**: drag-and-drop, либо `cp` / `copy`.
- **Удалённый прокси**: `scp ResourcePackManager-Velocity.jar user@proxy-host:/path/to/proxy/plugins/`
- **Хостинг-панель прокси** (Pterodactyl и т.п.): загрузка через файловый менеджер панели.

### 2. Перезапустите прокси

И всё. Никаких конфигов править не нужно, никаких ключей вставлять.

Плагин читает `plugins/floodgate/key.pem` на прокси при старте и автоматически
выводит из этого файла сетевую идентичность. Поскольку Floodgate и так требует,
чтобы этот файл был одинаковым на каждом бэкенде **и** на прокси (для работы
авторизации Bedrock), выведенный ключ автоматически совпадает с ключом
каждого бэкенда.

Первое слияние обычно завершается за ~10 секунд после того, как все бэкенды
поднимутся.

## Как проверить, что всё работает

**Консоль прокси** (в течение ~10с после перезапуска при условии, что бэкенды работают):

```
[ResourcePackManager] Merged pack ready at .../merged/Bedrock.zip (sha1 ...)
[ResourcePackManager] Geyser mappings deployed to .../custom_mappings
[ResourcePackManager] ✔ Network resource pack is now ready (... KB, sha1ABCD1234)
```

**Bedrock-клиент**: подключитесь через Bedrock. Перед входом в мир должен
появиться запрос на загрузку resource pack. Кастомные предметы рендерятся со
своими моделями вместо обычных стоек для брони.

**`/rspm status`** на бэкенде: показывает состояние пака, режим хостинга и
отпечаток network-key. Сравните последние 4 символа с конфигом прокси, чтобы
убедиться, что обе стороны связаны.

## Частые проблемы

### "Floodgate key.pem missing" при старте прокси

Плагин прокси не нашёл `plugins/floodgate/key.pem` и остался в режиме
ожидания. Решение:

1. **Установите Floodgate** на прокси. Он всё равно нужен, чтобы Bedrock-игроки
   могли достучаться до прокси, так что это обязательно вне зависимости от RSPM.
2. Убедитесь, что `plugins/floodgate/key.pem` на прокси **побайтно идентичен**
   тому же файлу на каждом бэкенде. Floodgate по умолчанию генерирует разные
   ключи для каждой установки — скопируйте один канонический `key.pem` с
   любого бэкенда на все остальные компоненты (другие бэкенды + прокси),
   затем перезапустите всё. Floodgate всё равно требует этого для авторизации
   Bedrock, так что если Bedrock-игроки в вашей сети сейчас работают, то это
   уже сделано.

### Bedrock подключается, но кастомных моделей не видно

Самая частая причина: прокси стартовал **раньше**, чем бэкенд успел собрать
свой первый Bedrock-пак. Geyser регистрирует кастомные предметы только при
старте — раз уж он запущен с пустым mappings-файлом, так и останется.
**Перезапустите прокси** после того, как бэкенд залогирует
`Wrote merged Geyser mappings: N entries`. Последующие слияния автоматически
подхватываются маршрутом раздачи пака в Geyser, но таблица кастомных предметов
фиксируется при старте.

### Предупреждения "Duplicate bedrock_identifier" при старте прокси

Два бэкенда выдали один и тот же Bedrock-идентификатор для одного и того же
базового предмета. Побеждает последний пишущий; безопасно, если вам нужен
лишь один бэкенд для подачи этого предмета. Если же оба бэкенда должны
держать **разные** кастомные предметы под одним базовым, у вас реальная
коллизия — переименуйте одну из исходных Java-моделей, чтобы автогенерируемые
хеши различались.

### Обновление RSPM

После обновления бэкенд-JAR RSPM также **повторно скопируйте** соответствующий
`ResourcePackManager-Velocity.jar` / `-BungeeCord.jar` из этой папки на прокси
и перезапустите прокси. Бэкенд перегенерирует их при каждом старте, поэтому
они всегда синхронизированы с версией бэкенда.
